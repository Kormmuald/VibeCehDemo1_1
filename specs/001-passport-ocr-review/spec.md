# Feature Specification: Распознавание и проверка паспорта

**Feature Branch**: `main`

**Created**: 2026-06-11

**Status**: Draft

**Input**: User description: "Upload одного PDF/PNG; recognition response с паспортными полями и возможным raw OCR text для ручной проверки; manual review перед confirm; обязательность passportSeries и passportNumber при confirm; допустимость пустых необязательных полей при partial OCR; Out of Scope: batch processing, cloud OCR, обязательный frontend, хранение исходных файлов и raw OCR text в БД."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Получение результата распознавания (Priority: P1)

Оператор отправляет один файл с разворотом российского паспорта в формате PDF или PNG и получает результат распознавания с паспортными полями, достаточный для последующей ручной проверки.

**Why this priority**: Без приема документа и выдачи распознанных данных невозможно выполнить основной пользовательский сценарий.

**Independent Test**: Отправить по одному поддерживаемому PDF и PNG и проверить, что для каждой попытки возвращаются статус обработки и полный набор паспортных полей, включая пустые значения для нераспознанных необязательных полей.

**Acceptance Scenarios**:

1. **Given** корректный PDF с одним паспортным документом, **When** оператор отправляет файл на распознавание, **Then** система возвращает отдельную попытку обработки и результат с паспортными полями для ручной проверки.
2. **Given** корректный PNG с одним паспортным документом, **When** оператор отправляет файл на распознавание, **Then** система возвращает тот же набор паспортных полей, что и для PDF.
3. **Given** документ распознан частично, **When** система формирует результат, **Then** надежно распознанные поля заполнены, неоднозначные или отсутствующие необязательные поля пусты, а попытка не считается технически неуспешной.
4. **Given** распознавание сформировало вспомогательный текст, **When** возвращается результат, **Then** ответ может содержать raw OCR text для ручной сверки без его постоянного сохранения.

---

### User Story 2 - Ручная проверка и подтверждение (Priority: P2)

Оператор просматривает распознанные значения, при необходимости исправляет их и явно подтверждает проверенный результат.

**Why this priority**: OCR не является источником окончательной истины; подтвержденная запись должна отражать решение человека.

**Independent Test**: Получить частичный результат, вручную заполнить или исправить значения и подтвердить его; затем проверить, что сохранены именно переданные при подтверждении данные.

**Acceptance Scenarios**:

1. **Given** результат распознавания доступен для проверки, **When** оператор исправляет значения и подтверждает их с непустыми `passportSeries` и `passportNumber`, **Then** система сохраняет подтвержденную запись с исправленными значениями.
2. **Given** результат содержит пустые необязательные поля, **When** оператор подтверждает его с заполненными серией и номером, **Then** подтверждение успешно, а необязательные поля остаются пустыми.
3. **Given** результат еще не был передан на ручную проверку, **When** предпринимается попытка подтвердить его автоматически, **Then** подтвержденная запись не создается.

---

### User Story 3 - Исправление ошибок подтверждения (Priority: P3)

Оператор получает понятную ошибку, если обязательные паспортные реквизиты отсутствуют или имеют недопустимый формат, и может исправить данные без повторного распознавания файла.

**Why this priority**: Ошибка в серии или номере делает подтвержденную запись непригодной, но не должна вынуждать повторно запускать OCR.

**Independent Test**: Последовательно отправить подтверждение без серии, без номера и с неверным форматом; убедиться, что запись не создается, а после исправления тех же данных подтверждение проходит.

**Acceptance Scenarios**:

1. **Given** результат доступен для ручной проверки, **When** оператор подтверждает его без `passportSeries` или без `passportNumber`, **Then** система отклоняет подтверждение, указывает обязательное поле и не создает подтвержденную запись.
2. **Given** серия или номер имеют недопустимый формат, **When** оператор отправляет подтверждение, **Then** система возвращает исправимую ошибку валидации и сохраняет возможность повторить подтверждение.

### Edge Cases

- За одну попытку передано более одного файла: попытка отклоняется до распознавания.
- Передан файл неподдерживаемого формата, пустой или нечитаемый файл: подтвержденная запись не создается, возвращается безопасное сообщение об ошибке.
- PDF не удается преобразовать в изображение либо обработка изображения/OCR завершается невосстановимым сбоем: попытка получает технически неуспешный статус.
- Документ читается плохо, но отдельные поля извлечены: возвращается частичный результат для ручной проверки, а не технический сбой.
- Серия и номер не распознаны: оператор может ввести их вручную перед подтверждением.
- Повторное подтверждение уже подтвержденной попытки не создает дублирующую запись и возвращает однозначный результат.
- Временные файлы остаются после прерванной или неуспешной обработки: они должны быть удалены после завершения попытки.
- OCR engine, language data, PDF rasterization, preprocessing или baseline-зоны отсутствуют либо отличаются от governing reference без отдельного решения: feature не считается готовой к реализации или приемке до устранения расхождения.
- `rus.traineddata` недоступен локальному OCR engine: попытка распознавания получает технически неуспешный статус с безопасным публичным сообщением, а система не переключается на другой OCR engine или cloud OCR.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST принимать ровно один файл PDF или PNG за одну попытку распознавания.
- **FR-002**: Система MUST отклонять неподдерживаемый формат, отсутствие файла и передачу нескольких файлов с понятным безопасным сообщением.
- **FR-003**: Система MUST создавать отдельную попытку обработки и различать как минимум результат для проверки, технический сбой, ошибку подтверждения и подтвержденное состояние.
- **FR-004**: Результат распознавания MUST содержать поля `passportSeries`, `passportNumber`, `lastName`, `firstName`, `middleName`, `birthDate`, `birthPlace`, `issueDate`, `divisionCode` и `issuedBy`.
- **FR-005**: Результат распознавания MAY содержать raw OCR text для ручной сверки в рамках ответа текущей попытки.
- **FR-006**: Система MUST оставлять пустым необязательное поле, если значение отсутствует, неоднозначно или не может быть надежно нормализовано.
- **FR-007**: Частично распознанный документ MUST быть доступен для ручной проверки и MUST NOT считаться техническим сбоем только из-за пустых необязательных полей.
- **FR-008**: Система MUST предоставлять оператору возможность передать исправленные значения всех паспортных полей при явном подтверждении результата.
- **FR-009**: Система MUST требовать отдельное подтверждение человеком после получения результата распознавания и MUST NOT автоматически создавать подтвержденную запись.
- **FR-010**: При подтверждении `passportSeries` MUST быть обязательным значением из 4 цифр, а `passportNumber` MUST быть обязательным значением из 6 цифр.
- **FR-011**: Все поля, кроме `passportSeries` и `passportNumber`, MAY быть пустыми при подтверждении.
- **FR-012**: При ошибке валидации подтверждения система MUST не создавать подтвержденную запись, MUST указать исправимое поле и MUST разрешить повторную отправку исправленных данных без повторного OCR.
- **FR-013**: После успешного подтверждения система MUST сохранять значения, переданные оператором при подтверждении, а не неизмененный первоначальный результат OCR.
- **FR-014**: Система MUST предотвращать создание нескольких подтвержденных записей для одной попытки обработки.
- **FR-015**: Невосстановимый сбой загрузки, преобразования документа, подготовки изображения или распознавания MUST переводить попытку в технически неуспешное состояние с безопасным публичным сообщением.
- **FR-016**: Исходный PDF/PNG, raw OCR text и промежуточные артефакты MUST NOT сохраняться постоянно и MUST удаляться после завершения обработки.
- **FR-017**: Система MUST исключать содержимое документа, raw OCR text и паспортные реквизиты из журналов.
- **FR-018**: Минимальная реализация MUST использовать локальный OCR baseline из governing reference: Tesseract, язык `rus` и файл `rus.traineddata`; замена OCR engine или подключение cloud OCR находятся вне scope.
- **FR-019**: Для PDF система MUST выполнять rasterization перед OCR с baseline 300 DPI, а PNG MUST направляться в тот же pipeline нормализации изображения, что и изображение, полученное из PDF.
- **FR-020**: Перед OCR система MUST выполнять preprocessing и normalization согласно governing reference, включая grayscale, contrast enhancement, thresholding variants, deskew attempt, document boundary detection, perspective/crop correction, conservative fallback и zone-specific preprocessing variants.
- **FR-021**: Распознавание MUST использовать zonal OCR model с baseline-зонами `issue`, `identity` и `number-vertical`, где `number-vertical` является источником `passportSeries` и `passportNumber`.
- **FR-022**: Baseline-координаты и padding зон из governing reference MUST NOT изменяться молча; любое изменение зон, preprocessing variants, thresholding strategy, deskew behavior или fallback behavior требует отдельного явно обоснованного решения, review и повторной проверки на sample documents.
- **FR-023**: Извлечение и нормализация полей MUST следовать governing reference: серия и номер извлекаются из candidate sequence из 10 цифр, даты приводятся к `dd.mm.yyyy`, `divisionCode` к `ddd-ddd`, текстовые поля проходят базовую нормализацию без автоматического угадывания неоднозначных значений.

### Governing References and Constraints *(mandatory)*

- **Reference**: `03-passport-ocr-reference-constraints.md`; `.specify/memory/constitution.md`.
- **Fixed OCR baseline**: Локальный Tesseract, язык `rus`, обязательный `rus.traineddata`, PDF rasterization с baseline 300 DPI, общий pipeline нормализации для PDF-derived images и PNG, preprocessing перед OCR, zonal OCR и ручная проверка до подтверждения.
- **Required preprocessing baseline**: grayscale, contrast enhancement, thresholding variants, deskew attempt, document boundary detection, perspective/crop correction, conservative fallback при ненадежном определении границ документа, full-page/padded-crop/deskew fallbacks и zone-specific preprocessing variants.
- **Baseline zones**: `issue` (`issuedBy`, `issueDate`, `divisionCode`), `identity` (`lastName`, `firstName`, `middleName`, `birthDate`, `birthPlace`) и `number-vertical` (`passportSeries`, `passportNumber`) с координатами и padding из governing reference.
- **Field normalization baseline**: `number-vertical` дает candidate sequence из 10 цифр, первые 4 цифры записываются в `passportSeries`, следующие 6 в `passportNumber`; даты нормализуются к `dd.mm.yyyy`, `divisionCode` к `ddd-ddd`, текстовые поля проходят trim, очистку шумных символов по краям и схлопывание повторяющихся пробелов.
- **Fixed product decisions**: Backend-first workflow, локальная обработка, частичный OCR как допустимый результат, ручная проверка до подтверждения, минимизация сохраняемых данных, исключение персональных данных из логов и запрет на автоматическое угадывание неоднозначных необязательных полей.
- **Permitted variations**: Параметры зон, preprocessing variants, thresholding strategy, deskew behavior, fallback behavior и правила нормализации отдельных полей могут изменяться только отдельным явно обоснованным решением после проверки на sample documents, review и обновления governing reference/technical plan.
- **Out of scope**: Batch processing, cloud OCR, обязательный frontend, authentication/authorization, автоматическое подтверждение, постоянное хранение исходных файлов, raw OCR text, crops, debug images и иных OCR-артефактов.

### Data Handling and Failure Modes *(mandatory when files or personal data are involved)*

- **Persisted data**: Подтвержденные оператором паспортные поля; минимальный идентификатор, статус и служебная metadata попытки; безопасное публичное сообщение для неуспешной попытки.
- **Temporary data**: Исходный файл, изображения страниц, зоны, raw OCR text и иные рабочие артефакты существуют только во время обработки и удаляются после ее завершения независимо от результата.
- **Logging exclusions**: Содержимое документов, raw OCR text, изображения, ФИО, даты, место рождения, серия, номер, код подразделения и сведения о выдаче.
- **Partial result**: Надежно распознанные значения возвращаются, необязательные неоднозначные значения остаются пустыми, результат доступен для ручной проверки.
- **Technical failure**: Попытка получает неуспешный статус; пользователю возвращается безопасное сообщение без внутренних диагностических данных и персональных значений.
- **Validation failure**: Подтвержденная запись не создается; оператор получает ошибки конкретных полей и может повторно подтвердить исправленные значения.

### Key Entities *(include if feature involves data)*

- **Processing Attempt**: Одна попытка обработки одного PDF или PNG; имеет идентификатор, статус, минимальную служебную metadata и при ошибке безопасное публичное сообщение.
- **Recognition Result**: Неподтвержденный набор паспортных полей для ручной проверки; может быть частичным и временно включать raw OCR text.
- **Confirmed Passport Record**: Итоговый набор значений после явной проверки оператором; всегда содержит серию и номер и может содержать пустые необязательные поля.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% проверочных сценариев один валидный PDF или PNG приводит либо к результату для ручной проверки, либо к однозначно обозначенному техническому сбою.
- **SC-002**: В 100% сценариев partial OCR пустые необязательные поля не блокируют ручную проверку и подтверждение при наличии корректных серии и номера.
- **SC-003**: В 100% сценариев подтверждение без корректной 4-значной серии или 6-значного номера отклоняется без создания записи.
- **SC-004**: В 100% сценариев сохраненная запись совпадает со значениями, которые оператор явно отправил при подтверждении.
- **SC-005**: После завершения каждой проверочной попытки исходный файл, raw OCR text и промежуточные артефакты отсутствуют в постоянном хранилище и журналах.
- **SC-006**: Оператор может пройти путь от получения результата до успешного подтверждения исправленных данных не более чем за 2 минуты, не считая времени самого распознавания.
- **SC-007**: Не менее 90% операторов в приемочном тестировании успешно различают частичный результат, технический сбой и ошибку подтверждения без дополнительной инструкции.
- **SC-008**: В 100% review-проверок перед переходом к plan зафиксирована трассировка к governing reference для OCR engine, `rus.traineddata`, PDF 300 DPI, preprocessing, zonal OCR, baseline-зон, правил нормализации, storage policy и failure behavior.

## Assumptions

- Оператор является доверенным пользователем внутреннего backend-first процесса; authentication/authorization рассматриваются отдельной feature.
- Один PDF представляет один паспортный документ; обработка нескольких документов или пакетная загрузка не поддерживаются.
- Raw OCR text является необязательным вспомогательным элементом recognition response и не требуется для успешного ручного подтверждения.
- Ручная проверка может выполняться любым клиентом контракта; создание отдельного frontend не является частью этой feature.
- Настоящая спецификация фиксирует обязательные OCR guardrails из governing reference, но детальные проектные решения, dependency versions, компоненты pipeline, тестовые fixtures и конкретная структура .NET solution должны быть раскрыты в technical plan.
